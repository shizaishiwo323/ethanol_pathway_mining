# AGENTS.md

本文件约束在本目录及其子目录中工作的 AI 编码/分析助手。

## 项目背景

本项目围绕“电化学 CO2/CO 还原生成乙醇”的文献挖掘与可视化分析展开，目标是从论文摘要、正文、图注和机理讨论中提取乙醇生成路径表述，并将不同文献中的路径说法归一到统一编码体系。

核心产出包括：

- 乙醇生成路径编码体系，例如 *CO 二聚、CO-CHO 偶联、CO-COH 偶联、CO-CHx 偶联、*OCHO/甲酸根相关路径、混合/未明确路径。
- 文献元数据表，记录 DOI、标题、年份、催化剂、电解液、反应类型、乙醇 FE、证据类型、提及路径、主路径和人工复核状态。
- 路径关键词词典，用于自动或半自动匹配机理相关句子。
- 频次统计和可视化结果，例如桑基图、气泡图、年份趋势图、催化剂-路径热图。
- 可复用的数据清洗、文本提取、统计分析和绘图脚本。

## 工作原则

### 先理解再动手

- 开始修改前，先阅读 `需求.md`、`计划.md` 和相关数据/脚本。
- 明确当前任务是在做文献检索、数据清洗、路径匹配、人工复核辅助、统计分析、可视化，还是报告整理。
- 如果任务描述存在多种解释，先写出假设；涉及研究范围、路径分类、统计口径或数据来源时，不要默默替用户决定。
- 对不确定的化学机理、文献结论、路径归类和 DOI 元数据，标注不确定性，不要伪造来源。

### 简单优先

- 先用最小可工作的流程解决当前问题，不为尚未确认的后续需求提前设计复杂框架。
- 不为单次使用代码创建过度抽象。
- 不添加未被要求的功能、配置项或“通用平台化”能力。
- 如果一个脚本 50 行能清楚完成任务，不要扩写成 200 行。

### 外科手术式修改

- 只修改与当前请求直接相关的文件和行。
- 不顺手重构无关代码、不改无关格式、不清理非本次变更造成的旧问题。
- 匹配项目现有风格；已有中文文档继续用中文，已有表字段命名风格继续沿用。
- 如果发现无关的死代码、重复数据或结构问题，可以在回复中指出，但不要擅自删除。
- 本次变更造成的无用导入、临时变量、中间文件引用，应在同次修改中清理。

### 目标驱动和可验证

对每个任务定义可检查的完成标准，例如：

- “增加路径关键词”应能验证：关键词文件格式正确，并能匹配一组示例句子。
- “清洗文献表”应能验证：DOI 去重结果、缺失字段数量、行数变化可追踪。
- “修复统计错误”应能验证：用最小样例复现错误，再确认统计输出正确。
- “生成图表”应能验证：图表文件存在、坐标/图例/标签可读、统计口径与数据表一致。

多步骤任务建议先写简短计划：

```text
1. 读取输入数据和字段说明 -> verify: 确认列名、行数、缺失值
2. 实现清洗或匹配逻辑 -> verify: 用小样例检查输出
3. 生成统计/图表 -> verify: 检查文件和关键数值
```

## 数据和文献处理约定

- 文献统计必须区分 `mentioned_pathways` 和 `main_pathway`，不要把综述性提及等同于论文主张。
- 路径归类需要保存证据句子，例如 `pathway_sentence` 或原文段落，方便人工复核。
- 对 `confidence`、`manual_checked`、`evidence_type` 等字段保持可追踪，不要用自动匹配结果冒充人工确认。
- DOI、标题、年份、期刊等元数据优先来自可靠数据源；自动补全的数据应保留来源或处理说明。
- 对 PDF 正文、摘要、图注和补充信息的提取要保留来源位置，例如 Abstract、Results、Discussion、Figure caption。
- 统计频次时应说明口径：按文献计数、按句子计数、按主路径计数，还是按提及路径计数。

## 代码和文件约定

- Python 分析脚本优先保持为可重复运行的脚本或 notebook，输入、输出路径写清楚。
- 表格字段使用稳定、可机器读取的英文列名，例如 `paper_id`、`doi`、`main_pathway`、`confidence`。
- 中文说明文档可以使用中文标题和解释，但数据列名、文件名尽量稳定简洁。
- 生成中间文件时，优先放入明确命名的输出目录，并在文档中说明用途。
- 不要覆盖原始数据文件；清洗后的数据另存为新文件，并保留从原始到清洗数据的转换逻辑。
- 涉及图表输出时，优先同时保留源数据表和图片文件，保证图可以复现。

## 安全删除规则

禁止批量删除文件或目录。

不要使用：

- `del /s`
- `rd /s`
- `rmdir /s`
- `Remove-Item -Recurse`
- `rm -rf`

需要删除文件时，只能一次删除一个明确路径的文件。

正确示例：

```powershell
Remove-Item "C:\path\to\file.txt"
```

如果需要批量删除文件，应停止操作，并请求用户手动删除或明确逐个确认。

## 验证习惯

- 修改代码后，运行与变更范围相匹配的检查，而不是只凭肉眼判断。
- 没有完整测试时，至少运行一个最小样例或脚本 dry run。
- 修改表格、JSON、CSV 后，检查文件能被正常读取。
- 修改可视化后，检查图文件是否生成，标签是否可读，统计数值是否和表格一致。
- 如果无法验证，最终回复中明确说明没有验证的部分和原因。

## 交流方式

- 回复用户时直接说明做了什么、改了哪些文件、如何验证。
- 对研究判断保持克制：能从数据或文献证据支持的结论才写成结论，推测要标明。
- 遇到项目范围、统计口径、机理分类存在争议时，先指出选择会影响结果，再给出建议。
- 不制造不存在的数据、文献、DOI、图表或运行结果。
